# 03-技术方案

## 1. 技术栈建议

- 前端：Next.js
- 后端：Next.js API Routes 或独立 Node 服务
- 数据库：SQLite 起步
- 图片存储：NAS 本地目录
- 队列：数据库任务表 + 后台 worker 轮询
- 部署：NAS Docker
- 模型：sub2api image-2

## 2. 架构原则

- 前端不直接持有 sub2api key
- 所有模型请求由后端代理
- 图片文件存在 NAS 本地目录
- 数据库只记录元数据、参数、任务状态和文件路径
- MVP 使用 SQLite，后续用量变大可迁移 Postgres

## 3. 目录建议

```text
huajing-studio/
├── app/                      # Next.js 页面
├── components/               # UI 组件
├── lib/
│   ├── db.ts                 # SQLite 连接
│   ├── sub2api.ts            # sub2api 调用封装
│   ├── storage.ts            # 本地图片存储
│   └── queue.ts              # 任务队列逻辑
├── workers/
│   └── image-worker.ts       # 后台生成任务 worker
├── data/
│   ├── app.db                # SQLite 数据库
│   └── images/               # 生成图片本地目录
├── docs/
└── docker-compose.yml
```

## 4. 任务队列设计

MVP 不引入 Redis / BullMQ，先用数据库任务表。

流程：

1. 前端提交生成请求
2. 后端写入 `generation_tasks`
3. worker 轮询 `queued` 任务
4. worker 调用 sub2api image-2
5. 下载/保存图片到 NAS 本地目录
6. 写入 `generated_images`
7. 更新任务状态为 `succeeded` 或 `failed`
8. 前端轮询任务状态并刷新结果

## 5. 图片存储

建议路径：

```text
/data/images/YYYY/MM/DD/<task_id>/<image_id>.png
```

数据库记录相对路径，前端通过后端静态文件接口访问。

## 6. 部署

部署位置：公司内网 / NAS。

建议：

- Docker Compose 部署
- 挂载本地 volume 保存 SQLite 和图片
- 环境变量保存 sub2api key
- 暴露一个内网端口访问

示例环境变量：

```env
SUB2API_BASE_URL=https://xxx/v1
SUB2API_API_KEY=***
IMAGE_STORAGE_DIR=/app/data/images
DATABASE_URL=file:/app/data/app.db
```

## 7. 后续可扩展

- SQLite → Postgres
- 本地存储 → S3 / R2
- DB 轮询队列 → Redis + BullMQ
- 单机 worker → 多 worker
- 增加访问控制 / 登录


## 8. image-2 接口固定调用配置

本项目 image-2 调用方式已在 `06-image-2接口调用说明.md` 写死。核心配置：

```env
SUB2API_BASE_URL=https://s2a.laolin.ai/v1
IMAGE_MODEL=gpt-image-2
```

后端 worker 调用规则：

- 文生图：`POST /v1/images/generations`
- 图生图 / 改图：`POST /v1/images/edits`

注意：图片生成请求建议带浏览器 User-Agent，并设置 180-900 秒超时。API Key 只允许存在服务端。
